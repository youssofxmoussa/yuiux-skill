# UX Error Prevention — Web Edition

This skill exists to make web software feel like it was built by people who have obsessed over every pixel, every millisecond, and every word — the standard set by the best software on Apple platforms, translated faithfully to the web. It is a working library, not a manifesto: every entry below and in the linked reference files states what's wrong, why it's wrong, and exactly how to fix it, usually with code.

It is organized as an **index** (this file) plus **deep reference files** in `reference/`. Read this file always. Read the specific reference file for whatever you're currently building. Read `reference/26-anti-pattern-catalog.md` when doing a full pre-launch UX audit — it is the exhaustive "every known UX error" list.

## How to Use This Skill

1. **While building**: before writing a form, nav bar, modal, table, chart, upload flow, or auth screen, skim the matching reference file's checklist first. It is cheaper to build it right than to retrofit it.
2. **While reviewing**: run the **Pre-Ship Gate** at the bottom of this file against the actual rendered UI (not just the code) before calling it done.
3. **When something "feels off" but you can't name it**: read `reference/01-philosophy-and-principles.md` — most vague "this feels cheap" complaints map to a violation of clarity, feedback, or consistency.
4. **When auditing an existing app**: work through `reference/26-anti-pattern-catalog.md` top to bottom; it's organized so you can grep for symptoms ("blank screen", "double submit", "tiny tap target") and jump straight to the fix.

This skill is platform-agnostic. Nothing here depends on Replit-specific tooling — the same rules apply if the surface is regenerated by another AI app builder (e.g. Lovable) or migrated between stacks. Code samples default to React + plain CSS/Tailwind-flavored class names because that is the most common target, but the underlying rule is always framework-independent; adapt the syntax, never the rule.

## The Apple Standard, Translated to Web

Apple's Human Interface Guidelines rest on three pillars — **clarity, deference, and depth** — plus a relentless obsession with **feedback** and **consistency**. None of these are iOS-specific; they are just usually easier to achieve inside a tightly controlled native SDK. On the web you have to earn them deliberately:

- **Clarity**: Every screen has one obvious primary action. Text is legible at every size the layout supports. Icons are never decorative alone — they always carry a label or a well-established meaning (settings gear, trash can). Nothing on screen exists without a reason a user could state back to you.
- **Deference**: Content is the interface. Chrome (borders, dividers, shadows, decorative gradients) recedes until it's needed to disambiguate. A UI shouldn't be memorable for its decoration; it should be memorable for how easy it made the task.
- **Depth**: Visual layers (elevation, blur, motion) communicate hierarchy and spatial relationships — a modal really is "in front of" the page, and the transition proves it. Depth is functional, not decorative: it's how a user understands what's temporary (a sheet) versus what's permanent (the page underneath).
- **Feedback**: Every action produces an immediate, proportionate response — a button press depresses, a save shows confirmation, a delete shows what was removed and how to undo it. Silence after an action is the single most common way software feels broken.
- **Consistency**: The same control looks and behaves the same everywhere in the app. A user should be able to predict a new screen's behavior from screens they've already used.

The rest of this skill operationalizes those five pillars into rules you can apply while writing code.

## The Core Rule

**Every interactive element must communicate three things at all times: what it is, what will happen if used, and what just happened.** Nearly every entry in this skill is a specific violation of one of those three, at a specific layer (visual, semantic, timing, or copy).

## Pre-Ship Gate

Before calling any UI work finished — before `presentArtifact`, before telling the user it's ready — walk the actual rendered app (not just the code) through this gate. Each row links to the reference file with the full detail and fixes.

- [ ] **Forms**: every input has a real label; errors name the field and the fix; input is never wiped on failed submit → `06-forms-inputs.md`
- [ ] **Controls**: no clickable `div`; every state (default/hover/focus/active/disabled/loading) is designed; hit targets ≥44px on touch → `07-buttons-controls.md`
- [ ] **Navigation**: current location always visible; every reachable state has a way forward; back button behaves correctly → `08-navigation-ia.md`
- [ ] **States**: loading/empty/error exist and are specific for every async view; no blank white screens; error boundary catches exceptions → `09-feedback-loading-empty-error-states.md`
- [ ] **Accessibility**: full keyboard operability; visible focus rings; WCAG AA contrast; screen-reader labels on icon-only controls → `10-accessibility-deep-dive.md`
- [ ] **Performance**: perceived feedback within 100ms of any action; heavy assets lazy-loaded; no layout shift on load → `11-performance-perceived-performance.md`
- [ ] **Responsive**: verified at mobile/tablet/desktop; no horizontal scroll; touch targets sized for fingers → `12-mobile-responsive-touch.md`
- [ ] **Motion**: transitions have purpose and respect `prefers-reduced-motion`; nothing animates just to animate → `05-motion-animation-microinteractions.md`
- [ ] **Copy**: no message blames the user; every error is actionable; terminology is consistent app-wide → `14-microcopy-content-strategy.md`
- [ ] **Destructive actions**: confirmed or undoable; nothing irreversible fires on one accidental click → `24-error-prevention-recovery.md`
- [ ] **Consistency**: the same component looks and behaves the same in every place it appears → `25-design-systems-consistency.md`

If any box is unchecked, that is a defect, not a nice-to-have — fix it before presenting the work.
