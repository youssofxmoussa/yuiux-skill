# UX Error Prevention

A practical, code-level checklist for avoiding the UX mistakes that make software feel broken or untrustworthy even when it "works." Apply this while writing UI code, not just at review time — most of these are cheaper to get right the first time than to retrofit.

This skill is platform-agnostic: the rules are plain HTML/CSS/JS/React patterns, so they apply whether the surface is built here, regenerated by another AI app builder (e.g. Lovable), or migrated between them.

## When to Use

- Building any form, navigation, list/detail view, auth flow, settings page, or file upload/download.
- Writing empty states, loading states, or error states.
- Before calling `presentArtifact` / declaring UI work done — run the **Pre-Ship Checklist** below.
- Reviewing or fixing an existing UI that "looks done" but hasn't been checked for these issues.

Skip the ceremony from the source material (CI pipelines, ML anomaly detection, dashboards, org rollout plans) — that's infrastructure for a dedicated UX tooling team, not something an agent building an app should set up unprompted. This skill keeps only the parts that change the code you write.

## The Core Rule

**Every interactive element must tell the user three things at all times: what it is, what will happen if used, and what just happened.** Nearly every UX error below is a violation of one of those three.

## 1. Forms & Inputs

- Every input has a real `<label>` (or visible text tied via `htmlFor`/`id`), never a placeholder standing in as the label — placeholders disappear on focus and fail screen readers.
- Mark required fields explicitly (`required` + visible indicator), and state format expectations before the user fails (`aria-describedby` hint text: "8+ characters", "YYYY-MM-DD"), not only after.
- Validate on blur/submit, not on every keystroke while a field is still incomplete — keystroke-level errors while typing feel hostile.
- Error messages name the field and the fix: "Password must be at least 8 characters," never "Invalid input" or "Error." Never blame the user.
- Preserve user input on validation failure or failed submit. Never clear a form because one field was wrong.
- Multi-step forms/wizards persist progress locally (state or storage) so a refresh or back-navigation doesn't discard work.
- Disable the submit button (or show a spinner) during submission, but keep it visible — don't hide it, and don't let it be double-clicked into duplicate submissions.

```html
<label for="email">Email</label>
<input id="email" type="email" required aria-describedby="emailHelp" />
<small id="emailHelp">We'll send a confirmation here.</small>
```

## 2. Buttons, Controls & Actions

- Button text describes the action, not the mechanism: "Save changes" / "Delete project", not "Submit" / "OK" for anything consequential.
- Use real `<button>`/`<a>` elements, never a clickable `<div>` or `<span>` — you get keyboard focus, Enter/Space activation, and screen reader semantics for free.
- Destructive actions (delete, remove, cancel subscription) require a confirmation step or an undo window — never fire on a single accidental click.
- Never rely on color alone to convey state (error red, success green) — pair it with an icon or text label for color-blind users.
- Disabled buttons should still be reachable by keyboard/screen reader with a reason surfaced (tooltip or nearby text), not silently unclickable.
- Every focusable element keeps a visible focus outline. If you override the default, replace it with an equally visible one — never `outline: none` with nothing in its place.

## 3. Navigation & Information Architecture

- Every page has a way back that doesn't require the browser back button (breadcrumb, back link, or persistent nav).
- Icon-only nav items (hamburger, gear, etc.) get an accessible label (`aria-label`) even if the visual label is omitted for space.
- The current location is always visually indicated (active nav state, page title, breadcrumb) — users should never wonder "where am I?"
- No dead ends: every state the user can reach (empty list, 404, error page) has at least one action that moves them forward (link home, retry, create first item).

## 4. Loading, Empty & Error States

These three states are usually an afterthought — treat them as first-class screens, not blank placeholders.

- **Loading:** show a skeleton or spinner immediately for anything that can take >300ms. Never leave a blank screen while data fetches.
- **Empty:** explain why it's empty and what to do next ("No projects yet — create your first one"), not just "No data" or a bare blank list.
- **Error:** be specific and actionable. "Couldn't save — check your connection and try again" beats "An error occurred." Always offer a retry action for recoverable errors (network, timeout) — never strand the user on a dead screen.
- Never let a JS exception produce a blank white screen. Wrap risky render trees in an error boundary that shows a friendly fallback with a reload/retry action.
- For optimistic UI updates (e.g. instantly showing a new item before the server confirms), roll back and show a clear error if the request actually fails — don't leave the UI lying about the true state.

## 5. Accessibility (non-negotiable baseline)

- Every image conveying information has meaningful `alt` text; purely decorative images use `alt=""`.
- Color contrast meets WCAG AA (4.5:1 for normal text, 3:1 for large text) — check this whenever picking a palette, not just for "gray on white."
- All interactive elements are reachable and operable by keyboard alone (Tab, Enter, Space, Esc for closing modals/dialogs).
- Modals/dialogs trap focus while open and return focus to the trigger element on close.
- Use semantic HTML first (`<nav>`, `<main>`, `<button>`, `<h1>`-`<h6>` in order) — reach for ARIA attributes only to fill gaps semantic HTML can't cover, not as the default.
- Respect `prefers-reduced-motion` for any non-essential animation.

## 6. Performance & Responsiveness

- Anything the user waits on for a perceptible duration gets immediate feedback (spinner, progress bar, skeleton) — perceived responsiveness matters more than raw speed.
- Lazy-load heavy assets (images below the fold, large charts, video) instead of blocking initial render.
- Debounce/throttle expensive handlers (search-as-you-type, resize, scroll listeners).
- Test at common breakpoints (mobile ~375px, tablet ~768px, desktop ~1280px+) — don't assume desktop-only usage. Tap targets on touch layouts should be at least ~44px.

## 7. File Uploads & Downloads

- Show upload/download progress for anything non-instant; never leave the user staring at a static button with no feedback.
- On upload failure, state what failed and offer retry — don't silently drop the file or reset the form.
- Validate file type/size client-side before upload starts, with a clear message on rejection (not a cryptic server error after the fact).

## 8. Microcopy & Tone

- Never phrase errors or empty states as if the user did something wrong. Describe the situation and the fix, not blame.
- Be consistent with terminology across the app — don't call the same object "item" in one place and "entry" in another.
- Confirmation/success messages should be specific ("Changes saved" beats a generic toast with no message).

## 9. Consent, Privacy & Auth UX

- Ask for permissions (location, camera, notifications) only at the point of use, with a one-line reason — never on first load with no context.
- Session expiry should be explained, not silent — "Your session expired, please sign in again" with a direct path back to what they were doing, not an unexplained redirect to a blank login page.
- Cookie/consent banners appear once, are dismissible, and don't block the whole page.

## 10. Internationalization (only if the app supports multiple locales)

- Don't hardcode text layout assumptions that break under longer translated strings or RTL languages (`dir="rtl"` support, mirrored icon direction where relevant).
- Use locale-aware date/number formatting instead of hardcoded formats.

## Pre-Ship Checklist

Run through this before presenting any UI work as done. Treat unchecked items as bugs, not polish:

- [ ] Every form input has a real label; errors name the field and the fix
- [ ] No action is a clickable `div`/`span`; keyboard-only navigation works end to end
- [ ] Destructive actions require confirmation or offer undo
- [ ] Focus outlines are visible on every interactive element
- [ ] Loading, empty, and error states exist and are specific — no blank screens
- [ ] Error boundary in place — a thrown exception shows a fallback, not a white screen
- [ ] Color is never the only signal for state (error/success/warning)
- [ ] Layout works at mobile, tablet, and desktop widths
- [ ] Every icon-only control has an accessible label
- [ ] Text contrast meets WCAG AA against its background
- [ ] Session/auth expiry and permission prompts explain themselves before acting

## Applying This With Existing Tooling

If the project already has ESLint configured, prefer adding `eslint-plugin-jsx-a11y` (React) for automated enforcement of the accessibility rules above rather than relying on manual review alone:

```json
{
  "extends": ["plugin:jsx-a11y/recommended"]
}
```

This is optional polish — the checklist above is the part that must not be skipped, regardless of what linting infrastructure the project has.
