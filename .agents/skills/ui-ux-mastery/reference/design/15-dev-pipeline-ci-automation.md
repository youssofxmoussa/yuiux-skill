# Dev Pipeline & CI/CD UX Automation

This file covers *process* — how to catch UX regressions automatically over time — complementing the rest of this skill's *content* guidance (what good UX looks like). Apply these patterns once a project has a real CI pipeline (most Replit projects should still do the manual pre-ship checklist in `00-priority-checklist.md` regardless of whether full CI automation is in place).

## Quality gate categories

A mature UX-quality pipeline runs four categories of automated checks on every PR/deploy, roughly in this order (fail fast on the cheapest checks first):

1. **Static analysis / lint** — accessibility linting (e.g. `eslint-plugin-jsx-a11y` for React, axe-core static rules) catches missing `alt`, missing form labels, invalid ARIA usage, and low-hanging contrast issues before a build even runs. Cheapest and fastest gate — should run on every commit, not just PRs.
2. **Automated accessibility audit** — run `axe-core`/`Lighthouse` accessibility audits against key rendered pages in CI (headless browser). Fail the build on any new "critical" or "serious" violation; track "moderate"/"minor" as a non-blocking report so the gate doesn't become so strict it gets disabled.
3. **Visual regression** — pixel-diff snapshot testing (e.g. Percy, Chromatic, Playwright screenshot comparison) on core screens/components. Catches unintended layout shifts, broken responsive breakpoints, and style regressions that unit tests can't see. Set an intentional diff-approval workflow (a human approves expected visual changes) rather than either ignoring diffs or blocking on every pixel change.
4. **Performance budget** — Lighthouse CI or similar, gating on Core Web Vitals thresholds (LCP, CLS, INP) so a feature PR can't silently regress load performance. Tie the specific budget to the product's line of business — a data-dense dashboard has different realistic targets than a marketing landing page.

## Real User Monitoring (RUM) feedback loop

Synthetic CI checks catch known regression classes; RUM catches what synthetic testing misses in production:

- Instrument Core Web Vitals (LCP, CLS, INP) and JS error rate from real user sessions, segmented by device class and network condition — aggregate lab-only metrics hide the P75/P95 mobile-on-slow-network experience that matters most.
- Track rage-clicks/dead-clicks (repeated rapid clicks on an element with no visible response) as a proxy for confusing or broken interactive elements — this surfaces UX bugs that don't throw errors and wouldn't be caught by any automated test.
- Feed RUM anomalies (a spike in a specific error, a CLS regression on a specific route after a deploy) back into the CI pipeline as a new regression test, closing the loop rather than treating each production incident as a one-off fix.

## AI-assisted UX testing and auto-remediation

- **AI visual review agents** can be run in CI against rendered screenshots, checking for the anti-patterns cataloged in `../errors/26-master-anti-pattern-catalog.md` (contrast failures, misaligned elements, inconsistent spacing) as a supplementary check alongside deterministic tools — useful for catching things pixel-diffing can't (e.g. "this looks unbalanced" rather than "this pixel changed").
- **Auto-PR generation for simple fixes** (missing `alt` text, an `aria-label` gap, a contrast-failing color token) can be scripted as a bot that opens a draft PR when the static/accessibility gate finds a fixable violation — keep these auto-fixes narrowly scoped to mechanical, low-risk changes; anything touching layout or interaction logic should stay a human-authored fix.
- Treat AI-generated UX suggestions as a *lead*, not an authoritative merge-blocking gate, unless the specific rule has been validated to have a very low false-positive rate in your codebase — an overly aggressive AI gate that blocks unrelated work erodes trust in the whole pipeline fast.

## Suggested pipeline order (fail fast, cheap-to-expensive)

```
lint (a11y + code style) → unit tests → accessibility audit (axe/Lighthouse)
   → visual regression → performance budget → (optional) AI visual review
   → deploy to staging → RUM monitoring feeds back into regression suite
```

## Platform-agnostic note

None of the above requires a specific host or IDE — the same gates apply whether the project lives on Replit, is generated by Lovable, or is built via Claude Code. The specific tool names above (axe-core, Lighthouse CI, Percy/Chromatic) are examples of the *category* of tool needed; substitute whatever's already available in the project's stack rather than introducing a new tool purely to match this reference.
