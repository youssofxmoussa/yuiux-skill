# AI Skill for UX Error Prevention – Developer Guide

## Executive Summary

This guide details the design of an **AI-powered UX error prevention skill** for web developers. It defines common UX error categories (forms, navigation, accessibility, etc.), outlines methods to detect and prevent them, and specifies how to integrate automated remediation and AI components into development workflows. The skill operates at all stages: in **IDEs and linters**, during **CI/CD pipelines**, and in **production monitoring**, providing real-time feedback, analytics, and suggestions. We leverage industry standards (e.g. WCAG, Nielsen’s heuristics) and official framework guidelines (React, Angular). Through static analysis (ESLint rules, axe scans), runtime telemetry (analytics, session replay), A/B testing, and ML/AI models for anomaly detection and code transformation, the skill continuously prevents regressions and “UX anti-patterns.” Preventive strategies (semantic HTML, validation, graceful degradation, optimistic UI updates) and *“shift-left”* reviews ensure issues are caught early. 

The roadmap covers an MVP (basic linting and reporting), through v1–v3 expansions (AI suggestions, real-time monitoring, multi-framework support). We provide code examples (lint config, CI YAML), sample API specs, and mermaid diagrams (architecture, workflows, timeline). Key metrics (conversion, error rates, performance) and governance (privacy, accessibility compliance) are tracked via dashboards. A/B tests and user studies validate improvements. This living reference is based on primary sources (W3C/WCAG standards, framework docs, seminal UX research) and showcases best practices with extensive citations.

## Definitions & Taxonomy of UX Errors

**UX errors** are design or implementation issues that impede usability or satisfaction. They span many categories:

- **Forms & Inputs:** Missing or unclear labels, poor validation, inaccessible form controls. *Example:* An `<input>` without `<label>` or `aria-label` violates WCAG (1.3.1) and frustrates users.  
- **Authentication & Security UX:** Unhelpful password requirements, confusing login flows, missing feedback (e.g. “caps lock on” warnings). Error messages blaming the user (e.g. "invalid input") break Nielsen’s heuristic “help users recover from errors”.
- **File Uploads/Downloads:** Lack of progress indicators or error recovery (e.g. upload failures with no retry). Errors during download can leave users uncertain.
- **Buttons & Controls:** Ambiguous text, disabled states, missing focus styling. E.g. relying on color alone for errors violates accessibility (color contrast guidelines).
- **Navigation & Information Architecture:** Deep or inconsistent menus, missing breadcrumbs, “hamburger menus” without labels. Overly complex navigation is a common UX pitfall.
- **Accessibility Issues:** Insufficient color contrast, missing alt text, non-semantic HTML, lack of keyboard focus cues. These violate WCAG and exclude users with disabilities.
- **Performance/Responsiveness:** Slow page loads, janky animations, unresponsive UI. Even 0.1s delays can measurably hurt conversion. Lack of mobile optimization also falls here.
- **Error States & Edge Cases:** Generic error pages (404/500), no clear recovery actions, or unusual corner cases (e.g. low disk space, offline scenarios) are neglected. Nielsen notes poorly placed or timed errors (e.g. toasts vs inline) confuse users.
- **Internationalization & Cultural UX:** Misplaced elements in RTL languages, untranslated text, local conventions (dates, number formats). For instance, Arabic UX demands mirroring layouts; failing to mirror button placement (e.g. “Login” on top-right) disrupts users.
- **Security-Related UX:** Overzealous captchas, unclear consent flows, broken links after authentication. UX must align with security (e.g. explain session timeouts rather than force logout silently).
- **Mobile vs Desktop Differences:** Inconsistent functionality (e.g. missing mobile navigation), tap targets too small on touchscreens.
- **Progressive Enhancement:** Sites that assume modern features (JS, cookies) may break for older browsers. A progressive enhancement approach provides basic functionality universally.
- **State Management (Offline/Real-time):** Failing to handle intermittent connectivity or updating stale data can confuse users. Offline-first apps should cache gracefully and queue actions.
- **Animations & Microcopy:** Over-fast or distracting animations (bad for motion-sensitive users), as well as unclear or blaming microcopy in prompts or button text, degrade UX.
- **Onboarding & Guidance:** Lack of tutorials or contextual help when a feature is first used.
- **Consent, Privacy & Legal:** Intrusive cookie banners, ambiguous consent forms, missing privacy notices can break trust and violate regulations.

Each category encompasses many common patterns (see table below for examples). By comprehensively defining these categories, the AI skill can check against a wide **taxonomy of UX anti-patterns** and map them to remediation strategies.

## Detection Methods

We employ **multimodal detection** to catch UX issues early and in production:

- **Static Analysis (Compile-Time):** Linting and code scans find bad patterns in code. For example, `eslint-plugin-jsx-a11y` statically checks React JSX for missing labels, alt text, tabindex issues, etc.. We use tools like ESLint/TSLint with accessibility plugins, CSS linters for responsive checks, and custom analyzers. This provides instant feedback in the IDE or CI.  
- **Runtime Monitoring:** Instrumenting the app to log user actions, errors, and performance. Tools like Sentry for JS errors, Chrome Error Collector, or Real User Monitoring (RUM) gather live UX data. Synthetic monitors (e.g. New Relic Synthetics, Datadog) emulate user flows to catch failures (dead links, load issues) after deployment.  
- **User Telemetry & Analytics:** Collect clickstreams, heatmaps, and session replay to identify friction. Aggregated heatmaps highlight “dead zones” or neglected elements, while session replays expose individual frustration (rage clicks, abrupt drop-offs). Standard analytics (Google Analytics, Mixpanel) track drop-out rates on forms, click-through rates, etc.  
- **A/B Testing & Experiments:** By serving different UI versions to segments, we statistically determine which performs better. For example, an A/B test might compare two error message formats; success is measured via conversion or task success. NN/g’s A/B Testing 101 emphasizes using quantitative metrics (conversion, retention, bounce) to validate UX changes.  
- **Heuristic Reviews & UX Audits:** Expert-driven evaluations using Jakob Nielsen’s heuristics (e.g. consistency, error recovery) and WCAG checklists. Heuristic analysis can catch issues not yet codified. For example, NN/g stresses clear, user-friendly error messages.  
- **ML-Based Anomaly Detection:** Advanced models detect unusual patterns (e.g. sudden spikes in 404 errors, or drops in form submissions). Natural language classifiers can group and interpret free-text user feedback or support tickets to surface UX pain points. While immature, AI techniques are included to flag non-obvious regressions.

## Prevention Strategies

Preventing UX errors involves applying proven design patterns and graceful degradation:

- **Semantic HTML & ARIA:** Use the correct HTML elements (e.g. `<button>` instead of clickable `<div>`) so browsers and assistive tools handle them natively. Provide ARIA labels only when necessary, and keep HTML semantics intact. For example, React docs emphasize that standard form controls *“need to be labeled accessibly”* to support screen readers.
- **Form Validation & Feedback:** Implement inline, real-time validation for error-prone inputs, but show errors only when meaningful (e.g. on blur or submit) to avoid “hostile patterns”. Provide clear, human-readable messages and suggestions (e.g. “Passwords must be 8+ characters” instead of generic “Invalid password”). This aligns with WCAG success criteria (error identification and suggestion).
- **Graceful Degradation & Progressive Enhancement:** Ensure a minimal but functional UX on older browsers or low connectivity. Use feature-detection (e.g. feature queries or `if ('serviceWorker' in navigator)`) and polyfills to progressively enable advanced features. Always offer basic content for everyone; e.g. ensure navigation works without JavaScript.
- **Fallbacks & Placeholders:** Provide alternative content for rich media (alt text, transcripts) and fallback fonts. For example, specify `alt=""` on decorative images (to avoid incomplete screen reader announcements) and provide meaningful alts where contentful. Use CSS media queries to adjust layouts on small screens so mobile UX isn’t broken.
- **Optimistic UI & Retries:** For network interactions (form submits, saves), optimistically update the UI (show expected result immediately) and then confirm with the server. If the request fails, display a clear error and revert if needed. Implement retry logic with exponential backoff for transient errors to avoid user frustration (e.g. retry image upload if upload fails).
- **Idempotent & Transactional UX:** Design multi-step workflows (checkouts, multi-page forms) so partial progress is saved. Allow users to resume interrupted tasks. For critical actions (payments, data deletion), use “confirm” dialogs or “undo” options to prevent accidental issues.
- **Consistency and Standards:** Follow platform UI conventions (e.g. Material Design, Apple HIG). Consistent button placement, color meaning, and terminology reduce user confusion. For instance, always use red for destructive actions and provide undo options.
- **Performance Budgets:** Enforce performance budgets (e.g. max 2s load) in code reviews. Use lazy loading for heavy resources, compress images, and minimize JS/CSS to avoid jank. Users expect quick feedback, and service workers can cache assets so pages load reliably offline.
- **Microcopy & Onboarding:** Use friendly, clear copy. Don’t blame the user – e.g. instead of “You did it wrong,” use “Please enter a valid email address”. Provide tooltips or short tutorials for complex flows. Ensure help text is available (`aria-describedby` linking to guidance). 
- **Privacy by Design:** By default, ask for minimal permissions and clearly explain why (e.g. ask for location only when needed). Comply with consent regulations (GDPR/CCPA) by obtaining clear opt-in and storing consent records.
  
Overall, the skill encodes these strategies as *rules* and *patterns*. For example, a lint rule might enforce `<label>` presence for inputs or detect use of inline `style="display:none"` (which can hide content from assistive tech). A UI component library can embed built-in validation and fallback behaviors.

```html
<!-- Accessible form example (JSX) -->
<form onSubmit={handleSubmit}>
  <label htmlFor="email">Email:</label>
  <input id="email" type="email" required aria-describedby="emailHelp" />
  <small id="emailHelp">Enter a valid email.</small>

  <label htmlFor="password">Password:</label>
  <input id="password" type="password" required aria-describedby="pwdHelp" />
  <small id="pwdHelp">8+ characters</small>

  <button type="submit">Log In</button>
</form>
```
*Listing: Properly labeled inputs with inline guidance (using `aria-describedby`) prevent form errors and support screen readers.*

## Automated Remediation

Our AI skill not only detects issues but **suggests or applies fixes**:

- **Code Suggestions:** In IDEs or pull requests, the tool can auto-generate code edits. For example, it can insert missing `alt` attributes or `aria-labels` based on context. The Aristek pipeline shows how an AI layer produced context-aware fix suggestions for common issues (missing labels, alt text). Developers review these suggestions in code review, greatly reducing manual work.
- **Lint Fixes & Code Transforms:** Static tools (ESLint, Prettier) can auto-fix simple patterns (e.g. convert `<div role="button">` to `<button>`). We can define code mod scripts to refactor patterns across a codebase.
- **CI/CD Gates:** Integrate checks into build pipelines so that critical issues block merges. For example, using axe-core in CI (via `axe-cli` or `@axe-core/playwright`) to fail on violations, or using Pa11y for end-to-end scans. The Aristek example enforced *“critical and serious violations”* to block merging, and surfaced moderate ones as warnings.
- **Lint Rules Repository:** Provide a shareable config (e.g. an NPM package) containing ESLint/TSLint rules and stylelint rules. For instance, a sample `.eslintrc.json`:
  
  ```json
  {
    "extends": ["plugin:jsx-a11y/recommended", "plugin:vuejs-accessibility/recommended"],
    "rules": {
      "jsx-a11y/alt-text": "error",
      "vuejs-accessibility/form-label": "error"
    }
  }
  ```
  *Listing: Example lint config enabling accessibility rules for React/Vue.*

- **CI Pipeline Example:** A GitHub Actions YAML might run tests and accessibility scans:
  
  ```yaml
  name: CI
  on: [push, pull_request]
  jobs:
    build-and-test:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v2
        - name: Install
          run: npm ci
        - name: Lint JS/TS
          run: npm run lint
        - name: Accessibility Audit
          run: npx axe-cli https://localhost:3000 --save artifact:report.json
        - name: Run Tests
          run: npm test
  ```
  *Listing: CI workflow with linting and an Axe accessibility audit. Critical violations (exit code) can be set to fail the build.*

The skill can automatically enforce best practices: e.g. ensure focus outlines are never removed without replacement, or that WCAG color-contrast ratios meet AA standards by warning on poor color combos. By embedding into CI/CD and IDEs, fixes become part of the normal review process.

## AI Components

Key AI-driven elements include:

- **Models & Data:** We train or fine-tune models on annotated UI data. For instance, a model could classify screenshots or CSS layouts as “likely to have contrast issues.” NLP models (like a fine-tuned transformer) could interpret user feedback or UI text to detect confusion. The Aristek case used an AI to triage and group violations, reducing noise. The same principle applies: an AI ranks findings by user impact.
- **Features & Labeling:** Training data could come from labeled UX issues (screen recordings, bug reports tagged by error type). Features might include DOM attributes, user event sequences, or image features. Datasets must cover diverse frameworks (React, Vue, Angular) and edge cases.
- **Evaluation Metrics:** We measure precision (false alarm rate) and recall (missed issues). For AI suggestions, user acceptance rates (how often developers accept an auto-fix) is a key metric. For runtime prediction (e.g. will user drop off), we track accuracy against real outcomes.
- **Bias & Safety:** Ensure models do not encode cultural or accessibility biases. For example, training only on English text UX would fail for other languages. We use diverse data and privacy-preserving techniques. Automated changes are always subject to human review for subjective aspects (keyboard focus flow, cognitive UX).
- **Privacy-Preserving Methods:** When analyzing user data (sessions, feedback), we anonymize or aggregate to comply with GDPR/CCPA. On-device computation (e.g. a VS Code plugin doing local linting without sending code to a server) can avoid data leaks. Embedding strong encryption and consent checks guards user privacy.

By combining rule-based methods with AI, the skill continuously learns. Over time, it can even predict UX issues from code patterns or user analytics, proactively warning teams before problems reach users.

## Integration Points & Developer Tools

The UX skill integrates throughout the development lifecycle:

- **IDE Plugins:** Extensions for VS Code/WebStorm can flag UX issues as you code (e.g. missing labels, poor color contrast). For example, an IDE plugin can use the same lint rules or call an AI model locally. Developers get immediate squiggles or hover explanations.
- **Version Control Hooks:** Pre-commit or pre-push hooks (e.g. Husky + lint) ensure quick feedback. 
- **CI/CD Pipelines:** Automated scans (lint, axe, style checks) run in Jenkins/GitHub Actions/GitLab. Blocking policies enforce a “zero critical UX errors” rule before merge.
- **Browser Extensions/DevTools:** A dev-mode browser extension could run audits on demand, or highlight focus/contrast issues directly in the DOM.
- **Runtime SDK:** Inject a small JS SDK into production builds to capture telemetry (clicks, errors) and send to the analytics backend. It can also run live checks (e.g. detect if high CLS/response times) and notify.
- **Server-Side Hooks:** On server-rendered pages or APIs, validate content output (e.g. ensure meta titles exist) or intercept missing translations. 
- **Analytics Pipeline:** Integrate with existing analytics (Google Analytics, Segment) to funnel UX metrics (form abandonments, click heatmaps).
- **APIs & CLI:** The skill exposes command-line tools (`ux-skill scan .`, `ux-skill audit http://staging`) and REST APIs (e.g. POST code to `/api/analyze` returns issues). Sample OpenAPI spec excerpt:

## Developer APIs & CLI

We provide both **CLI commands** and **programmatic APIs** so teams can integrate the skill as needed. 

- **CLI Tools:** Commands like `ux-skill check` for static analysis, `ux-skill monitor` for runtime data, and `ux-skill report` to generate dashboards. They can be installed via NPM (e.g. a `@company/ux-skill` package). For example:
  ```bash
  # Run static analysis on a project
  npx ux-skill check --path src/ --format json > report.json
  # Launch a lightweight server with live monitoring
  ux-skill run-server --port 4000
  ```
- **REST API:** A hosted or self-managed backend offering endpoints for deeper analysis (as illustrated above). Teams can post new PR diffs or user telemetry data to these APIs. Authentication (API keys, OAuth) secures access.
- **SDK for Code:** In addition to a CLI, a Node.js or Python library lets scripts invoke analysis functions directly. E.g. `analyzeUX(codeString)` returns issues.
- **Dashboard & CLI Alerts:** The CLI can also create visual reports. For instance, `ux-skill report` might compile metrics into a Markdown or HTML summary (or integrate with tools like Grafana via JSON outputs).

All developer-facing components are documented with sample requests, configuration options (yaml/json), and versioned changes. 

## UX Review & Human-in-the-Loop

Despite automation, human judgment is vital:

- **UX Review Process:** Establish a workflow where designers or UX specialists periodically review findings. For example, every release might include a UX checklist review, covering mobile layout, microcopy tone, and complex flows. 
- **Designated UX Owner:** Assign an “UX champion” on each team to triage alerts and guide fixes.
- **Combined Analyses:** Use automated results as input to design review sessions. E.g., heatmaps showing unexpected clicks trigger an investigatory meeting.
- **Heuristic Audits:** Schedule quarterly heuristic evaluations (using Nielsen’s 10 principles) to catch issues the AI might miss, like branding consistency or nuanced psychological cues.
- **User Testing Integration:** After deployment, run usability tests (moderated or unmoderated) and feed any discoveries back into the AI training set. Successful strategies from those tests can become new rules.

This **human-in-the-loop** ensures sensitive or subjective areas (color aesthetics, humor in copy) get expert oversight. It also empowers teams to override or refine AI suggestions, continuously improving model quality.

## Testing & QA

We rigorously test the skill and the product with:

- **Unit Tests:** Validate individual UI components and utility functions. E.g. ensure a custom `<Button>` component always sets `type="button"` or that form validator catches missing fields. Tools: Jest, Mocha, Vue Test Utils, React Testing Library.
- **Integration Tests:** Test interaction of multiple components (e.g. filling a form and checking submission behavior). E.g. with React Testing Library or Angular TestBed.
- **End-to-End (E2E) Tests:** Simulate user flows in the browser. Use Cypress or Selenium to test real scenarios (login, checkout), including network failure scenarios. E2E suites can include accessibility checks (with `cypress-axe`, for instance).
- **Synthetic Monitoring:** Automated bots periodically run critical user journeys in production (login, add-to-cart) and alert on failures or performance regressions. 
- **Performance & Load Tests:** Use tools like Lighthouse CI for performance budgets, and load testing tools (k6, JMeter) to ensure UI stays responsive under scale.
- **Chaos Testing:** Intentionally inject failures (slow network, API errors) to verify fallbacks. For example, use a service like Gremlin or custom scripts to cut off backend and ensure the UI shows a friendly offline message rather than breaking.
- **Accessibility Regression Testing:** Integrate tools like `axe-core` into tests to automatically fail if a new change violates WCAG rules. For example, snapshot tests can enforce that certain ARIA attributes exist.

We maintain **synthetic metrics dashboards** (e.g. Lighthouse scores over time, a11y violation count) and **CI coverage thresholds**. Any pull request that decreases critical UX metrics or introduces new failures is flagged.  

**Checklist for UX QA:**  
- All interactive elements are keyboard-accessible and have focus indicators.  
- Critical paths have tests (e.g. user can complete checkout).  
- WCAG criteria (1.1.1, 3.3.x etc.) all pass automated tests.  
- Real user metrics (CPU, memory, load time) are within targets.

## Observability & Dashboards

We continuously monitor UX post-deployment:

- **Key Metrics:** Page load times (LCP, FID, CLS), API response times, error rates (JavaScript exceptions per session), form conversion rates, bounce rates, user engagement (time on task). Tools like Google Analytics, Datadog RUM, or Datadog Session Replay provide these.
- **Alerts:** Set thresholds (e.g. if 404 errors >1% of requests or FCP >3s) to trigger alerts via email/Slack. Sentry/Datadog can send alerts on crash loops or anomalies.
- **Session Replay & Heatmaps:** As described earlier, use session replay (FullStory, LogRocket) to watch real user sessions. Session replay uncovers “why” behind metrics: a sudden drop in form submissions might coincide with a modal failing to appear.
- **Dashboards:** A central dashboard (Grafana or Kibana) aggregates:
  - **UX Health Score:** Composite of accessibility compliance (% of WCAG passed), performance scores (Lighthouse), and user satisfaction (survey NPS).
  - **Component Metrics:** For a component library, track usage and issues per component.
  - **Issue Trends:** Chart of issues found/solved over time.
- **Logging:** All AI-skill findings and decisions are logged for audit. For example, if AI auto-fixes code, logs record which rule was applied.

These observability measures tie back to UX. For instance, if error messages are improved, we should see drop in form abandonment. By correlating UX error fixes with metric improvements, we validate the skill’s impact.

## Governance, Compliance & Ethics

Our solution ensures compliance and ethical use:

- **Standards Compliance:** We align with W3C and WCAG standards as authoritative references. For example, form validation techniques reference WCAG 3.3.1 and 3.3.3 requirements. Meeting legal requirements (e.g. European Accessibility Act, ADA) is a goal; the skill reports on these compliance levels.
- **Privacy:** As noted, user data collection follows GDPR/CCPA best practices. Consent banners (cookie pop-ups) are standardized and tested to ensure no required action is hidden. Data anonymization and strict storage policies (e.g. encrypted, short retention) are enforced. The skill itself never exfiltrates code — all analysis can run on-premises or in user-controlled CI, protecting IP.
- **Ethical Considerations:** The AI skill avoids bias by not penalizing culturally relevant designs. For example, it won’t flag right-to-left mirroring as “inconsistent.” We document ethical constraints: any automated change is vetted to avoid cultural insensitivity. Privacy by Design means minimal user tracking and opt-in analytics.
- **Security-related UX:** Ensure the skill’s own integrations are secure (e.g. OAuth for APIs, sandboxed execution of any code analysis). If the skill suggests UI changes, it avoids insecure defaults (e.g. never weakens auth or inserts insecure cookies).

A simple **privacy impact template** (see below) is included so teams can assess the UX skill’s data flows. Security audits of the tool (dependency scans, pen-testing of any web interface) are regular.

## Performance & Scalability

The AI skill itself must scale:

- **Performance:** Static analysis and AI inference should be fast. We can parallelize checks (multi-threading, caching). CI jobs can split by directory. The IDE plugin runs incrementally.
- **Scalability:** In enterprise use, the system may serve many projects. We recommend a microservices architecture (see Architecture diagram) so components like the analysis server or ML service can scale independently. A cloud deployment (Kubernetes, serverless functions) can auto-scale on load.
- **Resource Planning:** Large codebases might take minutes to analyze; we support scheduling (nightly runs) and priority settings.
- **High Availability:** Since the skill may block releases, ensure the service has redundancy and health checks. CI-integrated tools should fail safely (e.g. if the AI service is down, fall back to basic linting).

While optimizing UX is critical, we also measure the overhead: e.g., the added CI runtime or bundle size of the analytics SDK. Budgets and performance checks (e.g. Lighthouse) are built-in to the skill.

## Localization & Cultural UX

The skill addresses international and cultural nuances:

- **i18n Best Practices:** It checks for untranslated text (missing `i18n` wrappers), incorrect locale formats, and layout issues (text overflow in other languages). For RTL support, it ensures the `dir="rtl"` attribute is used where needed, and that mirrored layouts exist.
- **Cultural Sensitivity Checks:** The skill can flag culturally-specific imagery or idioms if they may not be appropriate globally. (Teams can configure cultural context; e.g. a Spanish site might warn if English text appears).
- **Locale Testing:** Automated tests include running the app in different locales. A/B tests or user studies must include diverse user groups. 
- **Language & Copy:** Verify microcopy for clarity; if localized copy is unclear (e.g. due to machine translation), encourage review by native speakers.
- **Accessibility Across Cultures:** Ensure color uses (e.g. red/green meaning) consider color-blind users universally.

Citing the MENA example, ignoring RTL can *“confuse users and disrupt overall experience”*. Thus, the skill specifically checks UX flows for correct adaptation (mirrored button order, proper text direction, culturally appropriate iconography).

Key elements:

- **LocalDev:** IDE plugin and CLI give instant feedback (section *Integration Points*). 
- **Version Control/CI:** The central pipeline runs static analysis (e.g. ESLint, axe) and tests. An AI engine can process results (clustering, suggestion generation) and update issues in Git (e.g. as PR comments).  
- **Production:** The live front-end/back-end is instrumented. A small JS SDK collects user experience data (clicks, errors) into an analytics database. Dashboards visualize real-time UX health (error trends, session replays).  
- **Feedback Loop:** Insights from production (heatmaps, errors) feed back into the dev process (issues are created for unresolved UX problems).

**Deployment Patterns:** The skill can be deployed as:
- A **cloud service** (SaaS) accessed via APIs and CLI (suitable for teams not wanting self-hosting).
- **On-premise** for enterprises (Docker/K8s images for static analysis service, ML inference server, analytics DB).
- **Embedded libraries** (npm packages and Docker images) integrate into CI.

## Acceptance Criteria & Evaluation Checklist

To claim “prevents UX errors,” the skill must meet objective criteria:

- **Coverage:** All known UX categories have at least one check (forms, navigation, accessibility, performance, etc.).  
- **Accuracy:** >95% of flagged issues are true positives (low false alarms), as measured by a validation set.  
- **Resolution Rate:** >80% of identified issues are either auto-fixed or corrected by developers before release.  
- **No Regression:** Introducing the skill must not increase build times or CI failures unrelated to UX.  
- **Performance Impact:** The runtime SDK adds <2% overhead to page load (measured by before/after).
- **UX Metrics Improvement:** Post-deployment, target improvements (e.g. 10% reduction in form drop-off, or WCAG compliance at AA) in pilot projects.  

**Evaluation Checklist:** (to be used by teams)
- ✅ **Form Fields:** All inputs have associated `<label>` or `aria-label` (WCAG 3.3.1).  
- ✅ **Error Messages:** All user-visible errors are clear, actionable, and non-technical.  
- ✅ **Navigation:** Key tasks reachable in <3 clicks; no dead-end pages.  
- ✅ **Accessibility:** Passes automated WCAG AA scans on core pages. Keyboard navigation intact (tab order, focus indicators).  
- ✅ **Performance:** Largest Contentful Paint <2.5s on mid-tier devices; no unresponsive script warnings.  
- ✅ **Mobile:** UI adapts correctly at common breakpoints (responsive checks).  
- ✅ **Localization:** UI displays correctly in all supported locales (dates, RTL languages).  
- ✅ **Privacy/Security UX:** Cookie banner appears only once and allows granular consent.  
- ✅ **Error Handling:** Network or server errors show user-friendly messages with retry options (no blank screens).  

The tool integrates into QA sign-offs: releases only proceed if critical checklist items are green.

## Sample User Study & A/B Test Design

To validate improvements, conduct the following **A/B experiment**:

1. **Participants:** 200 users (100 per group), matched on demographics.  
2. **Task:** Complete a multi-step signup form and checkout.  
3. **Baseline (A):** Original UI. **Variant (B):** UI with AI-skill fixes (enhanced validation, better labels, etc.).  
4. **Metrics:** Form completion rate, time on task, error count (number of validation errors encountered), and satisfaction (post-task survey on ease-of-use, 1–5).  
5. **Hypothesis:** Variant B will have ≥10% higher completion, fewer errors, and higher satisfaction.  
6. **Significance:** Use χ² test for conversion rate and t-test for task time/satisfaction (p<0.05).  
7. **Data Collection:** Instrument both versions to log all interactions. Consider heatmaps/session replay on variant B to confirm where users click.  

Additionally, a **usability study** can be run: have 8 users perform key tasks on variant B (with think-aloud) and record qualitative issues. This complements A/B’s quantitative data.  

By analyzing these studies, we confirm the AI skill’s UX fixes truly help users. For example, if users in B rarely use the “Help” link (indicating they didn’t need it), this suggests error messages were sufficiently clear.
